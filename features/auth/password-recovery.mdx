---
title: "Password recovery"
description: "Two mechanisms for recovering account access in SIUT-ITSM: a magic link that signs the user in directly, and a token-based flow that requires setting a new password first."
---

SIUT-ITSM provides two complementary recovery paths. Both are built on top of the same `password_resets` table and share the same token-generation policy (`MagicLinkTokenPolicy`), but differ in what happens after the token is consumed.

## `password_resets` table

Temporary recovery tokens are stored in the `password_resets` table:

```sql
CREATE TABLE IF NOT EXISTS password_resets (
    email      VARCHAR(255) NOT NULL,
    token      binary(16)   NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX (email),
    INDEX (token)
);
```

Tokens are binary(16) values (raw bytes). `MagicLinkTokenPolicy` validates the token format and `PasswordRecoveryInterface::findEmailByValidToken()` checks for existence within a **15-minute TTL** (`MagicLinkTtlMinutes = 15`). A token is consumed (deleted) as soon as it is used successfully.

## Initiating recovery

Both flows start the same way: the user submits their email on the recovery page at `/cuentas/recuperar-contra.php`. `PasswordResetUseCase` handles this request.

```php
// PasswordResetUseCase::execute()
$token = $this->magicLinkTokenPolicy->generate();
$this->passwordRecoveryRepository->storeMagicLink($email, $token);
$magicLink = $this->urlBuilder->to("/cuentas/recuperar-contra.php", [
    "token" => $token,
]);
$this->passwordRecoveryNotifier->sendMagicLink($email, $magicLink);
```

If the email does not correspond to any existing account, `PasswordResetUseCase` returns silently without sending an email — the response to the end user is identical to the success case to prevent account enumeration.

<Info>
  A `password_reset` event is written to `auth_logs` regardless of whether the email matched an account.
</Info>

## Recovery flows

<Tabs>
  <Tab title="Magic link (sign in directly)">
    The magic link flow authenticates the user immediately when they click the link, without asking them to choose a new password. It is handled by `RecoverPasswordWithMagicLinkUseCase`.

    <Steps>
      <Step title="User requests recovery">
        The user enters their email on `/cuentas/recuperar-contra.php`. `PasswordResetUseCase` generates a token, stores it in `password_resets`, and emails the magic link.
      </Step>
      <Step title="User clicks the link">
        The email contains a URL of the form:
        ```
        /cuentas/recuperar-contra.php?token=<hex>
        ```
        The browser opens this URL and the server calls `RecoverPasswordWithMagicLinkUseCase::execute(token)`.
      </Step>
      <Step title="Token validation">
        `MagicLinkTokenPolicy::isValid()` checks the token format. `PasswordRecoveryInterface::findEmailByValidToken(token, ttlMinutes: 15)` looks up the associated email. If the token is missing, malformed, or expired, a `password_reset` failure event is logged and the flow stops.
      </Step>
      <Step title="Token consumption">
        `PasswordRecoveryInterface::consumeToken()` deletes the token from `password_resets`, preventing replay.
      </Step>
      <Step title="Session created">
        `UserContextInterface::set()` stores an `AuthenticatedUser` built from the user's credential into the session. The user is now signed in without ever typing a password.
      </Step>
      <Step title="Success logged">
        `AuthEventLogger::passwordResetSuccess()` writes a successful `password_reset` event to `auth_logs`.
      </Step>
    </Steps>
  </Tab>

  <Tab title="Token-based (change password)">
    The token-based flow requires the user to choose a new password before being signed in. It is handled by `ChangePasswordWithTokenUseCase`.

    <Steps>
      <Step title="User requests recovery">
        Same initial step: the user submits their email, `PasswordResetUseCase` stores a token and sends the link to `/cuentas/recuperar-contra.php?token=<hex>`.
      </Step>
      <Step title="Token validation">
        `ChangePasswordWithTokenUseCase::validateToken(token)` is called first to verify the token is valid and not expired (15-minute TTL) before showing the password-change form. If invalid, `null` is returned.
      </Step>
      <Step title="User submits new password">
        The user fills in and submits the new password form. The server calls `ChangePasswordWithTokenUseCase::execute(token, newPassword)`.
      </Step>
      <Step title="Password hashed">
        PHP's `password_hash($newPassword, PASSWORD_DEFAULT)` produces a new bcrypt hash.
      </Step>
      <Step title="Atomic update">
        Inside a database transaction (`TransactionManager::transactional()`), both operations happen atomically:
        1. `CredentialRepositoryInterface::updatePassword(userId, passwordHash)` updates the stored hash.
        2. `PasswordRecoveryInterface::consumeToken(token)` deletes the used token.
      </Step>
      <Step title="Session created">
        `UserContextInterface::set()` stores the `AuthenticatedUser` in the session. The user is now signed in.
      </Step>
      <Step title="Success logged">
        `AuthEventLogger::passwordResetSuccess()` writes a successful `password_reset` event to `auth_logs`.
      </Step>
    </Steps>

    <Note>
      If the token is valid but the associated email no longer matches an active user, the token is consumed and a `passwordResetFailed` event is written — the stale token cannot be replayed.
    </Note>
  </Tab>
</Tabs>

## Failure conditions

Both use cases call `AuthEventLogger::passwordResetFailed()` (logged as `success = false` in `auth_logs`) when:

- The token string fails `MagicLinkTokenPolicy::isValid()` — malformed token.
- `findEmailByValidToken()` returns `null` — token not found or older than 15 minutes.
- A valid token resolves to an email with no matching user record.

<Warning>
  Tokens expire 15 minutes after creation. After expiry, the user must request a new recovery email.
</Warning>
